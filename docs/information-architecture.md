# The Establish Beauty — 情報設計（説明書）

このドキュメントは、ブランド・サイト・本リポジトリの役割を整理したものです。実装の前提として参照してください。

## ビジネスモデル（要約）

自社で**大量在庫を持たない**。**ブランドを集め**、**EC 導線**を提供し、**販売手数料**（取引連動のレベニュー）でプラットフォームを運営する。

## 実装イメージ：マルチベンダー型マーケットプレイス

実務上の型は **マルチベンダー型マーケットプレイス** と捉える。**見せ方は LP ごとに分け、買い手の体験はマーケット単位で統合する**（現代的な型）。

### ブランド専用 LP（世界観を分ける）

各ベンダーは**専用ランディング**を持ち、トーン・ビジュアル・ストーリーを分離する。URL は例として次のような **パスプレフィックス** を想定する（スラッグは契約・実装で確定）。

| 例（パス） | 役割 |
|------------|------|
| `/brands/redlight` | ブランド A の世界観・商品 LP |
| `/brands/kbeauty` | ブランド B の世界観・商品 LP |
| `/brands/organic` | ブランド C の世界観・商品 LP |

### 買い手側は統合（アカウント・カート・チェックアウト）

| 要素 | 方針 |
|------|------|
| **アカウント** | **共通**（単一の顧客プロファイル／ログイン体験。マーケット横断で同一アカウント）。 |
| **カート** | **共通**（マーケット全体で 1 つのカート）。 |
| **Checkout** | **共通**（**統合チェックアウト** — 一度の決済フロー）。 |
| **決済** | 上記チェックアウトに集約（**セントラライズド／まとめ決済**）。 |
| **フルフィル** | **分割**（**split fulfillment** — 注文行・ベンダーごとに出荷・在庫は各社）。 |
| **注文データ** | **各会社へ分配**（ピッキング・追跡・サポートの責務に応じてベンダーへ）。 |

在庫・出荷の主担当は原則パートナー側とし、プラットフォームは**集積・LP 基盤・統合チェックアウト・分配・手数料**を担う想定。

### 英語でのモデル説明（ピッチ・資料用）

実質にかなり近い言い方の例：

1. **Multi-vendor beauty marketplace with unified checkout and split fulfillment.**
2. **Curated beauty commerce platform with centralized checkout and distributed vendor fulfillment.**

オーディエンス提示用の一文（従来）：

> A curated beauty marketplace connecting beauty device manufacturers, D2C skincare brands, and online-first beauty companies.

## ポジショニング：Beauty Commerce Infrastructure

単なる **EC サイト（単店舗）** ではなく、**Beauty Commerce Infrastructure（美容コマースのインフラ）** に寄せた構造とする。以下を**ひとつの基盤で統合**することで、プラットフォーム側の価値を大きくする。

- **ブランド集積**
- **LP 提供**（ブランドごとの世界観・導線）
- **決済**（統合チェックアウト）
- **顧客導線**（発見〜購入までの一貫体験）
- **注文分配**（ベンダーへのオーダー振り分け）
- **顧客獲得**（マーケット／キュレーションに紐づく獲得）

## 構造ラベル（3 つの柱）

コピー・プロダクト・ピッチで共通して使う階層名。

| ラベル | 意味 |
|--------|------|
| **Marketplace** | 複数ブランドを横断した「市場」— 発見から購入導線まで。 |
| **Curated Commerce** | 掲載基準と編集によるキュレーションされたコマース体験。 |
| **Inventory-light Platform** | 自社フルフィル・大量在庫を前提とせず、パートナー在庫・物流と役割分担する基盤。 |
| **Beauty Commerce Infrastructure** | 上記を束ねる総称。**単なる EC ではなく**、ブランド集積・LP 提供・決済・顧客導線・注文分配・顧客獲得を**一つの基盤で統合**するレイヤー。 |

## ドメインとブランド

| 要素 | 役割 |
|------|------|
| **theestablishbeauty.com** | 親ブランドのドメイン。本リポジトリでは **`/`（`index.html`）を EMR-TEK 公式ストアのホーム**とし、親ブランドの短い入口は `establish-beauty.html`、旧ルートにあった長文の会社・インフラ説明は **`about-us.html`（私たちについて）** に置いている。`/emr-tek.html` は `/` へリダイレクトする。 |
| **Beauty Marketplace** | **マルチベンダー型**のマーケットレイヤー。共通カート・まとめ決済・注文のベンダー分配とセットで設計する。 |
| **EMR-TEK** | The Establish Beauty の**取引先のひとつ**。マーケットプレイス上のベンダー／ブランドの一つとして位置づける。 |

## 本リポジトリ（`redlight`）の位置づけ

- **EMR-TEK を前面にした静的サイト**として運用する（ルートはストア）。
- **既存の全 HTML ページ・構成はそのまま残す。** ヘッダー・フッターが EMR-TEK 前提のままでよい。
- メインサイトの**コンテンツのたたき台**は `establish-beauty.html`（親ブランドのホーム、Marketplace／取引先の説明・EMR-TEK への導線）。マーケットプレイス全体のシェルや本番 URL のルート割り当ては、別デプロイ・別構成でもよい。

## 将来のサイト構造イメージ（参考）

メイン・マーケットを別途立ち上げる際の URL イメージ例。実装時に調整してよい。

- `/` … Establish Beauty メイン（Beauty Commerce Infrastructure のハブ）
- `/marketplace` または `/shop` … 横断一覧・検索（**共通アカウント・共通カート・共通チェックアウト**の入口のひとつ）
- `/brands/redlight`、`/brands/kbeauty`、`/brands/organic` … **ブランド専用 LP**（世界観分離。スラッグは例）
- `/brands/{vendor}` … 一般形。例：EMR-TEK は本リポジトリの商品・コレクション HTML が現状の LP 相当。将来は同パターンの URL へ寄せる想定
- 注文確定後 … **split fulfillment** — 注文行ごとにベンダーへ分配（API・管理画面・メール等は実装時に定義）
